## Description
`push_swap` is a classic 42 algorithmic puzzle. The problem statement is simple: you get two stacks (`A` and `B`). Stack `A` starts with a chaotic pile of unique random numbers, and Stack `B` starts empty. Using a highly restricted set of operations (swapping the top two elements, shifting the whole stack up or down, or pushing elements between stacks), you have to sort Stack `A` in ascending order. 

The real catch? You have to do it using the absolute minimum number of moves possible. It’s an exercise in data structures, time complexity optimization, and bitwise manipulation.

## How I Built It: My Step-by-Step Logic

I didn't just jump into sorting algorithms straight away. I broke the architecture down into isolated milestones to ensure the core was completely unbreakable before optimizing the moves.

### Milestone 1: The Data Structure & Infrastructure
I chose a **Doubly Linked List** to represent the stacks. Since the project requires frequent shifting from the top to the bottom (and vice versa) via rotation commands, a doubly linked list lets me update pointers at $O(1)$ efficiency without constantly shifting arrays in memory. I built `lst_add_bottom`, `lst_new_node`, and memory tracking structures early on.

### Milestone 2: Robust Validation & String Tokenization
Before parsing anything, the input data must be bulletproof. I built a validation layer that checks for:
* Non-numeric parameters.
* Duplicate integers.
* Integer overflows and underflows.

The parser handles standard multiple arguments (`./push_swap 3 2 1`) as well as single-string quoted inputs (`./push_swap "3 2 1"`) by dynamically splitting strings using a customized pointer array handler (`free_split`).

### Milestone 3: Coordinate Compression
Instead of trying to sort massive or negative numbers directly (which messes up bit-shifting later), I implemented a ranking pre-pass. The program analyzes the raw inputs and overwrites their indices with a simplified range from `0` to `size - 1`. For example, `[84, -5, 42]` scales down neatly to `[2, 0, 1]`. 

### Milestone 4: The Small Sort Engine ($\le 5$ elements)
Automated algorithms like Radix are overkill for tiny stacks and waste too many moves. 
* **3 Elements:** I mapped out the exact 5 unsorted permutations on paper and built a hardcoded decision tree that sorts them in a maximum of 2 moves.
* **4 & 5 Elements:** The program finds the absolute smallest index node, shifts it to the top using the shortest path (`ra` or `rra`), pushes it to Stack `B`, recursively sorts the remaining 3 elements using the small engine, and drops them back down.

### Milestone 5: The Large Sort Engine (Radix Sort)
For stacks containing 100 to 500 elements, I used a bitwise **Radix Sort**. Because the numbers are already compressed into clean indices, the engine looks at their binary bits from right to left (Least Significant Bit to Most Significant Bit). 
* It loops through each bit column.
* If a node has a `0` bit, it pushes it to Stack `B`. 
* If it has a `1` bit, it keeps it in Stack `A` via a rotation (`ra`).
* Once a bit level is completed, it pulls everything back from Stack `B` and shifts to the next bit mask. This scales elegantly well within 42's efficiency thresholds.

## Instructions

### Compilation
The project handles its own compilation, including building the internal `libft` dependency automatically. Run the following command at the repository root:

make

To wipe the binary and force a clean re-compile from scratch:

make re

### Execution
Pass your list of integers directly into the executable:

./push_swap 23 54 1 89 14 6 9

Verifying with the 42 Checker
You can pass the output of push_swap into the official Linux checker tool to verify that the operations leave the stack sorted and empty out Stack B:

Basic validation
./push_swap 3 2 1 | ./checker_linux 3 2 1

Testing large random sets
ARG="23 54 1 89 14 6 9"; ./push_swap $ARG | ./checker_linux $ARG

To view the raw count of operations generated by the sorting engine:

ARG="5 3 1 4 2"; ./push_swap $ARG | wc -l
